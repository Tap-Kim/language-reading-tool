# 영문 아티클 읽기 흐름 교정 크롬 플러그인 설계 문서

## 1. 문서 목적

이 문서는 웹 페이지에서 사용자가 선택한 영문 텍스트를 대상으로, 읽기 흐름을 방해하는 요소를 즉시 시각적으로 교정하고, 교정 전후를 화면에서 바로 비교할 수 있는 크롬 플러그인의 기능·구조·작업 범위를 정의한다.

목표는 단순 번역이나 단어 뜻 제공이 아니라, 아래의 읽기 보조 경험을 즉시 제공하는 것이다.

- 문장을 의미 단위로 묶어 보여주기
- 끊어 읽기 위치를 시각적으로 드러내기
- 수식 관계를 덜 헷갈리게 재배치해서 보여주기
- 중요한 구조를 강조하여 한 번에 문장 골격을 잡게 만들기
- 원문을 훼손하지 않고 오버레이 방식으로 결과를 즉시 비교하게 만들기
- 모르는 단어와 문장을 드래그해 개인 메모로 저장하기
- 저장한 단어/문장과 해석 결과를 우측 접이식 패널에서 누적 관리하기

---

## 2. 제품 한 줄 정의

사용자가 웹에서 선택한 영어 문장과 모르는 단어를, 더 자연스럽게 읽히도록 교정하고 해석·메모까지 우측 패널에서 관리하게 해주는 인라인 리딩 코치형 크롬 확장 프로그램.

---

## 3. 해결하려는 핵심 문제

영문 아티클을 읽을 때 사용자가 느끼는 문제는 크게 세 가지다.

1. 단어는 알아도 문장의 흐름이 한 번에 잡히지 않는다.
2. 어디서 끊어 읽어야 하는지 감이 없어 매 문장에서 다시 해석한다.
3. 문장 구조를 눈으로 빠르게 파악하지 못해 읽기 속도와 이해가 동시에 떨어진다.

따라서 이 플러그인은 "해석 결과"보다 "읽기 방식의 교정"에 집중해야 한다.

---

## 4. 핵심 사용자 시나리오

### 4.1 기본 시나리오

1. 사용자가 웹 페이지에서 영문 문단 또는 문장을 드래그 선택한다.
2. 선택 직후 플로팅 액션 바가 나타난다.
3. 사용자가 원하는 교정 기능 또는 메모 저장 기능을 선택한다.
4. 화면에서 선택 영역이 즉시 교정된 형태로 오버레이 표시되거나, 선택 단어/문장이 우측 패널에 저장된다.
5. 사용자는 원문/교정본 비교, 강조 레벨 조절, 해석 확인, 메모 접기/펼치기를 수행한다.
6. 만족스러우면 결과를 고정, 복사, 다시 분석하거나 해제한다.

### 4.2 확장 시나리오

- 전체 문단에 의미 덩어리 슬래시 표시 적용
- 긴 문장에서 주절/종속절 분리 강조
- 전치사구·관계절·삽입구를 시각적으로 흐리게 처리
- 주어/동사/목적어 골격 강조
- 문장 난이도 또는 막힘 포인트 힌트 제공
- 오디오 리듬 연습용 끊어 읽기 뷰 제공
- 모르는 단어를 단어장처럼 우측 패널에 누적 저장
- 저장된 문장별 해석과 사용자 메모를 접고 펴서 확인

## 5. MVP 범위

MVP는 "선택 영역 즉시 교정"과 "비교 가능한 시각화"에만 집중한다.

### 5.1 포함 기능

- 선택 영역 감지
- 플로팅 툴바 표시
- 교정 요청 실행
- 인라인 오버레이 렌더링
- 원문/교정본 토글
- 교정 옵션 전환
- 결과 복사
- 단어/문장 메모 저장
- 우측 패널 열기/닫기
- 해제 및 원상복구

### 5.2 제외 기능

- 로그인
- 사용자 학습 기록 클라우드 동기화

- PDF/이미지 OCR
- 모바일 브라우저 지원

## 6. 기능 설계

## 6.1 선택 영역 인식

### 목적
사용자가 웹 페이지에서 선택한 실제 텍스트 범위를 기준으로 동작해야 한다.

### 요구사항
- mouseup 시점에 selection 존재 여부 확인
- 선택 문자열이 영어 중심인지 간단한 휴리스틱으로 판별
- 최소 길이 조건 적용, 예: 20자 이상 또는 3단어 이상
- input, textarea, code, pre, contenteditable 영역 여부 구분
- 너무 넓은 선택 범위는 경고 또는 축약 처리

### 예외 처리
- 선택 영역이 비어 있음
- 지나치게 긴 범위
- DOM이 복잡해 Range 계산이 불안정한 사이트
- Shadow DOM 내부 요소
- iframe 내부 문서

---

## 6.2 플로팅 액션 바

## 6.2A 메모 저장 액션

### 목적
사용자가 모르는 단어나 이해가 안 되는 문장을 읽는 흐름을 끊지 않고 즉시 저장해두도록 한다.

### 액션 정의
- 단어 저장
- 문장 저장
- 해석 추가 저장
- 나중에 보기

### 동작 원칙
- 드래그 후 `메모 저장`을 누르면 우측 패널에 즉시 항목이 추가된다.
- 단어 선택 시에는 기본적으로 단어 뜻 중심, 문장 선택 시에는 문장 해석 중심 카드가 생성된다.
- 같은 항목을 다시 저장하면 중복 생성 대신 누적 횟수 또는 최근 저장 시각을 갱신한다.

---

## 6.2B 우측 사이드 패널

### 목적
저장한 단어와 문장을 웹 페이지를 벗어나지 않고 같은 화면 우측에서 계속 참조하게 한다.

### 패널 구성
- 패널 헤더: 열기/닫기, 전체 접기/펼치기, 검색, 정렬
- 섹션 탭: 단어, 문장, 전체
- 카드 목록: 원문, 해석, 사용자 메모, 출처 페이지, 저장 시각
- 액션: 핀, 수정, 삭제, 복사

### 인터랙션
- 기본 상태는 접힘 가능 패널이며, 사용자가 열면 마지막 폭을 유지한다.
- 각 카드도 개별적으로 접고 펼칠 수 있어 긴 문장 해석을 한 번에 다 보지 않아도 된다.
- 패널은 페이지 스크롤과 독립된 자체 스크롤 영역을 가진다.

### 카드 예시 필드
- 원문 단어 또는 문장
- 품사 또는 구조 힌트
- 한국어 해석
- 한 줄 사용자 메모
- 저장 위치 URL 또는 페이지 제목
- 반복 저장 횟수

---


### 목적
선택 직후 사용자의 의도를 최소 클릭으로 이어준다.

### 버튼 제안
- 흐름 교정
- 의미 덩어리
- 구조 강조
- 간소화 보기
- 메모 저장
- 원문 비교
- 복사
- 닫기

### UX 원칙
- 선택 영역 근처에 위치하되 뷰포트 밖으로 나가지 않음
- 첫 클릭까지 빠르게 반응해야 하므로 애니메이션은 150ms 내외
- 키보드 접근 가능
- 사이트 스타일과 충돌하지 않도록 Shadow DOM 또는 isolated root 사용

## 6.3 교정 모드 정의

플러그인의 핵심은 여러 교정 모드를 제공하되, 결과 표현은 일관되게 유지하는 것이다.

### A. 흐름 교정 모드
문장을 실제 읽기 순서에 맞게 덜 버겁게 보이도록 시각 구조를 재정렬한다.

예시 효과:
- 긴 수식구 흐리기
- 핵심절 진하게 표시
- 끊어 읽기 지점에 간격 또는 약한 구분선 추가
- 종속절을 단계적으로 들여쓰기 또는 톤 다운

### B. 의미 덩어리 모드
문장을 slash chunk 또는 segment block 형태로 분절해 의미 단위로 읽게 돕는다.

예시:
- The tool / helps readers / process long English sentences / more naturally.

### C. 구조 강조 모드
주어, 동사, 목적어, 보어, 전치사구 등 핵심 구조를 색상 또는 배지로 구분한다.

예시:
- S: The tool
- V: helps
- O: readers
- Modifier: process long English sentences more naturally

### D. 간소화 보기 모드
삽입구, 부가설명, 관계절 등 이해를 방해하는 부위를 접거나 흐리게 하여 골격만 먼저 읽게 한다.

### E. 비교 모드
원문과 교정본을 상하 또는 좌우로 나란히 보여준다.

---

## 6.4 즉시 반영 방식

### 렌더링 원칙
- 원문 DOM을 직접 수정하지 않는다.
- 선택 영역 위에 비파괴적 오버레이를 렌더링한다.
- 해제 시 원문 상태가 완전히 복원되어야 한다.

### 렌더링 방식 후보

| 방식 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 원문 DOM 직접 치환 | 구현이 단순해 보임 | 사이트 깨짐, undo 어려움, 충돌 위험 큼 | 비추천 |
| 절대 위치 오버레이 | 원문 보존, 제어 쉬움 | Range 계산 복잡 | 추천 |
| 사이드 패널 표시 | 구현 쉬움 | 즉시 비교감이 약함 | 보조 기능용 |
| 툴팁/팝오버만 사용 | 가볍다 | 문장 전체 교정에는 한계 | 부분 보조용 |

### 최종 권장
- 1차는 선택 영역 기준 오버레이 렌더링
- 2차는 우측 사이드 패널로 상세 분석과 메모 누적 보강

---

## 6.5 교정 결과의 시각 규칙

### 시각 계층
- 핵심 골격: 진한 텍스트
- 덜 중요한 수식: 연한 텍스트
- 끊어 읽기 지점: 슬래시 또는 소프트 세퍼레이터
- 문장 단위: 카드형 배경
- 구조 라벨: 작은 배지 또는 색상 underline

### 원칙
- 색상보다 타이포 그래피 계층을 우선 사용
- 원문 가독성을 해치지 않도록 과도한 하이라이트 금지
- 한 화면에 정보 레이어는 최대 3개까지만 노출
- 디폴트는 가장 단순한 뷰여야 함

---

## 6.6 결과 후처리 액션

- 원문 보기
- 교정 보기
- 비교 보기

- 구조 강조 켜기/끄기

- 복사: plain text / markdown 스타일
- 다시 분석
- 피드백 보내기

## 7. 정보 구조

### 7.1 확장 프로그램 구성 요소

- Content Script: 선택 감지, 오버레이 렌더링, 이벤트 처리
- Background Service Worker: 메시지 중계, 컨텍스트 메뉴, 설정 관리
- Popup UI: 전역 설정, 단축키 안내, 실험 기능 토글
- Options Page: 고급 설정
- Right Sidebar Panel: 저장된 단어/문장/해석 렌더링
- Analysis Engine: 교정 규칙 처리 또는 API 호출 추상화
- Storage Layer: 로컬 설정 및 메모 저장

### 7.2 메시지 흐름

1. Content Script가 selection 추출
2. 교정 요청 payload 생성
3. Background 또는 local engine에 전달
4. 분석 결과 수신
5. Content Script가 오버레이 렌더링
6. 필요 시 메모 저장 및 우측 패널 갱신
7. 사용자 상호작용 이벤트 반영

## 8. 분석 엔진 설계

MVP 단계에서는 분석 엔진을 교체 가능한 구조로 설계해야 한다.

### 8.1 엔진 전략

#### 전략 A. 로컬 규칙 기반
- 장점: 빠름, 비용 없음, 개인정보 노출 적음
- 단점: 문장 구조 분석 정확도 한계

적합 용도:
- 간단한 chunk 분리
- 쉼표/접속사/전치사 기반 1차 분절
- 너무 긴 수식구 탐지

#### 전략 B. 원격 LLM/API 기반
- 장점: 문맥 기반 교정 품질 높음
- 단점: 응답 지연, 비용, 개인정보 이슈

적합 용도:
- 구조 강조
- 읽기 흐름 교정 문장 재표현
- 종속절/삽입구 구분

#### 전략 C. 하이브리드
- 로컬 전처리 후, 고품질 교정이 필요할 때만 원격 호출

### 8.2 권장안

초기 버전은 하이브리드가 가장 현실적이다.

- 1차: 로컬에서 선택 텍스트 검증, 언어 판별, 문장 분리
- 2차: 필요 시 원격 분석 요청
- 3차: 렌더링용 구조화 JSON 반환

### 8.3 반환 데이터 스키마 예시

```json
{
  "sourceText": "The tool helps readers process long English sentences more naturally.",
  "mode": "flow_rewrite",
  "segments": [
    { "text": "The tool", "role": "subject", "priority": "high" },
    { "text": "helps", "role": "verb", "priority": "high" },
    { "text": "readers", "role": "object", "priority": "high" },
    { "text": "process long English sentences", "role": "complement", "priority": "medium" },
    { "text": "more naturally.", "role": "adverbial", "priority": "low" }
  ],
  "chunks": ["The tool", "helps readers", "process long English sentences", "more naturally"],
  "collapsedRanges": [],
  "tips": [
    "먼저 주어와 동사를 잡고 읽기",
    "마지막 부사구는 뒤에서 가볍게 처리"
  ]
}
```

## 9. UX 상세 설계

## 9.1 첫 사용 경험

첫 실행 시 3단계 온보딩만 제공한다.

1. 영어 문장을 드래그 선택하세요.
2. 원하는 교정 모드를 누르세요.
3. 원문과 교정본을 비교하세요.

### 첫 사용 시 반드시 보여줄 것
- 이 확장은 페이지 내용을 영구 수정하지 않음
- 민감한 페이지에서는 비활성화 가능
- 단축키로 빠르게 실행 가능

---

## 9.2 인터랙션 원칙

- 기본 흐름은 마우스 2클릭 이내
- 오버레이는 ESC로 즉시 닫힘
- 다시 선택하면 기존 오버레이 자동 정리
- 사이트 스크롤 시 오버레이 위치 동기화
- 페이지 리사이즈 시 재계산

## 10. 기술 아키텍처

## 9.4 메모 데이터 흐름

1. 사용자가 단어 또는 문장을 선택한다.
2. 플로팅 액션 바에서 `메모 저장` 또는 `해석 후 저장`을 선택한다.
3. 분석 엔진이 단어 뜻 또는 문장 해석을 생성한다.
4. 저장 레이어가 메모 항목을 생성하거나 기존 항목을 갱신한다.
5. 우측 패널이 즉시 다시 렌더링된다.
6. 사용자는 카드 단위로 접기/펼치기, 수정, 삭제를 수행한다.

### 메모 저장 단위
- word: 단어 단위 저장
- sentence: 문장 단위 저장
- mixed: 짧은 구문 저장

## 14. 성능 기준

MVP에서 중요한 것은 "즉시 반응한다"는 인상이다.

### 목표
- 선택 감지 후 툴바 표시: 100ms 이내 체감
- 로컬 교정: 150ms 이내
- 원격 교정: 1.5초 이내 목표
- 오버레이 렌더링: 1 frame 내 반영

### 최적화 포인트
- 긴 텍스트는 문장 단위로 분할
- 레이아웃 계산 최소화
- 스크롤 이벤트는 throttle
- 잦은 분석 요청은 debounce
- 동일 텍스트 재요청은 메모리 캐시

## 24. 최소 프로토타입 정의

프로토타입 단계에서 반드시 확인할 것은 "선택 → 클릭 → 바로 더 읽기 쉬워 보임"과 "선택 → 저장 → 우측에서 다시 참조 가능함"의 성립 여부다.

### 프로토타입 성공 기준
- 임의의 기사 문장 선택 후 3초 내 결과 표시
- 사용자 스스로 교정 전보다 골격 파악이 쉽다고 느낌
- 저장한 단어와 문장을 우측 패널에서 바로 다시 찾을 수 있음
- 한 번 사용 후 바로 이해 가능한 조작 구조
- 사이트가 깨지지 않음

### 프로토타입 실패 신호
- 결과가 원문보다 더 복잡해 보임
- 오버레이 위치가 자주 틀어짐
- 툴바가 거슬림
- 분석 품질보다 반응 속도가 먼저 불만으로 나옴
- 메모가 쌓일수록 패널 탐색이 어려워짐

## 26. 최종 판단

이 제품은 번역 도구가 아니라 읽기 흐름 교정 도구로 포지셔닝해야 차별점이 분명해진다.

초기 성공 여부는 분석 모델의 고도화보다도 아래 두 가지에 달려 있다.

- 선택 직후 결과가 매우 빨리 나오는가
- 교정 결과가 실제로 더 쉽게 읽히는가

따라서 첫 버전은 "복잡한 AI 기능"보다 "빠르고 안정적인 선택형 오버레이 경험"을 우선 구현하는 것이 맞다.
